# 大模型展示模块规划

## 目标定位

当前“大模型展示”不应只是一个通用聊天窗口，而应成为 MovieRec 的智能电影助手入口。它需要围绕现有电影、影人、评论、画像和可视化数据，展示大模型能为电影推荐系统带来的增值能力。

## 核心用户场景

1. 电影问答
   - 查询电影剧情、类型、主演、导演、上映年份、地区等基础信息。
   - 支持追问，例如“这部电影还有哪些相似作品？”。

2. 推荐解释
   - 用户输入偏好后，模型给出推荐理由。
   - 能解释首页推荐、画像推荐或搜索推荐为什么出现某些电影。

3. 影人分析
   - 汇总某位影人的代表作品、常见合作对象、类型倾向。
   - 支持“这个演员适合推荐给喜欢什么电影的人？”。

4. 评论洞察
   - 总结某部电影评论区的高频观点。
   - 区分正面、负面和争议点。

5. 数据可视化解读
   - 对图表查询结果做自然语言解释。
   - 支持“为什么某一年某类型电影更多？”这类探索式问题。

## 页面信息架构

建议将现有聊天页重构为三栏或两栏布局：

- 左侧：会话历史和快捷入口。
- 中间：对话主区域，支持 Markdown、引用来源、加载状态。
- 右侧：上下文面板，展示本次回答引用的电影、影人、评论或图表数据。

移动端可以隐藏右侧上下文面板，将引用内容折叠到回答下方。

## 功能分区

1. 智能问答
   - 保留通用对话能力。
   - 默认提示应围绕电影推荐，而不是泛问答。

2. 电影知识库
   - 从数据库检索电影、影人、评论。
   - 回答中展示数据来源，例如电影名、影人名、评论条数。

3. 推荐助手
   - 用户输入偏好，例如“想看节奏快的日本悬疑片”。
   - 返回推荐列表、推荐理由、可点击跳转的电影卡片。

4. 图表解读
   - 对已有图表模块的查询结果进行总结。
   - 输出趋势、异常点、可能解释。

## 前端需求

- 模式切换：智能问答、知识库、推荐助手、图表解读。
- Prompt 卡片：空状态下提供 4 到 6 个可点击问题。
- 流式回答：支持模型逐字输出，提升响应体感。
- 停止生成：请求过慢或答非所问时可中止。
- 重新生成：对最后一条回答重新请求。
- 引用区：回答下方展示相关电影/影人/评论来源。
- 会话标题：根据第一条用户问题自动生成。
- 错误态：区分网络失败、模型失败、数据库无结果。

## 后端需求

- `POST /api/chat`
  - 非流式兼容接口，保留当前调用方式。

- `POST /api/chat/stream`
  - 流式输出接口，用于前端实时渲染。

- `POST /api/ai/retrieve`
  - 根据问题召回电影、影人、评论等结构化上下文。

- `POST /api/ai/recommend`
  - 将自然语言偏好转成推荐查询条件，并返回推荐结果和解释。

- 模型配置
  - API Key、模型名、base URL 必须放入环境变量。
  - 支持本地开发 fallback，模型不可用时返回清晰错误。

## 数据上下文设计

回答前应先构造结构化上下文：

- movie：电影名、类型、地区、年份、评分、简介、主演、导演。
- person：姓名、职业、简介、代表作品。
- comment：评论内容、用户、时间、电影 ID。
- chart：图表类型、筛选条件、聚合结果。

模型回答时优先使用这些结构化数据，避免凭空编造。

## 推荐实现顺序

1. 重做空状态和模式切换，让页面目标更清楚。
2. 增加 Prompt 卡片和右侧上下文面板。
3. 后端补检索接口，把数据库结果注入模型上下文。
4. 增加流式输出、停止生成、重新生成。
5. 接入推荐解释和图表解读。
6. 持久化会话历史。

## 验收标准

- 用户能在空页面直接知道可以问什么。
- 每种模式至少有 2 个可点击示例问题。
- 知识库回答能展示引用的电影或影人来源。
- 推荐助手能返回可跳转的电影卡片。
- 模型服务不可用时不会白屏，能给出明确错误。
- 前端在桌面和移动端都不会出现输入框、消息、引用卡片重叠。
